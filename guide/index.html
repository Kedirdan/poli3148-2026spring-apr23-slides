Put your webpage
on the web.

A complete student guide to publishing your project to the internet using GitHub Pages and an AI coding assistant.

Kaidierdan Muzhapaer · TA Spring 2026 Slide deck →

This guide combines everything from the lab session — the slides, the spoken explanations, the prompts, and the troubleshooting notes — into one readable document. You can read it top to bottom the first time, and come back later for the specific parts you need.

By the end, you should be able to take any webpage you have on your computer and publish it online with a public URL that anyone can open. You do not need to write a single Git command yourself. Your AI assistant (GitHub Copilot) handles all of that for you.

Today's goal

The lab is a test to make sure you can use GitHub Pages. As long as you end up with a live URL showing your webpage on the internet, you've done what's needed. Once you are comfortable with this flow, those of you who want to can use the same method to submit Assignment 1 later — but that is for another day, not today.

A working deployment needs your operating system, your AI assistant, and GitHub to cooperate closely. Any small mismatch between them can surface as an error, and these are the kinds of things we work through one at a time. If you hit something that doesn't click on the first try, that is entirely normal.

Before you start

You need four things set up beforehand:

  • A GitHub account. Sign up at github.com/signup with your HKU email.
  • GitHub Student Developer Pack, which includes free GitHub Pro. Apply at education.github.com/pack.
  • VS Code, installed on your laptop. Download from code.visualstudio.com.
  • GitHub Copilot, enabled in VS Code and signed in with your GitHub account. (Free for students.)

If any of these are missing, set them up first. Everything below assumes all four are ready.

Part 1

Concepts you should understand

Before we publish anything, let's get four terms straight: Git, GitHub, .gitignore, and GitHub Pages. If you already know them, skim this part and jump to Setup.

Git

Git is a tool that manages different versions of your files, especially code files. It runs on your computer, without internet.

Imagine you are writing a short description of HKU. You write version one. You add a word, change a verb, add a fact. You save each time. Without Git, you end up with files called draft 1, draft 2, final, and final version 2. You have probably done something like this before.

With Git, every save becomes a commit — a labeled snapshot of your project at that moment. Each commit has a name (technically a long hash like a1b2c3, but we simplify to v1, v2, v3, v4 for teaching) and a message, a short note of what changed.

And here is the part that matters. All four versions are still there, on your computer, at the same time. You can always go back to any one of them.

GitHub

So what is GitHub?

Git runs on your computer. You can use it even if you never share anything, and it still remembers every version you saved. But when you want to put your code online, or work with other people, you need a cloud platform. GitHub is the biggest open-source community and code-hosting platform in the world. That's why we use it in this class.

Code moves between your computer and GitHub in two directions. You upload it with git push, and download it with git pull. Git does not sync automatically — it is entirely up to you, or up to your AI assistant, to decide when to sync.

Note

In this lab, you never type git push or git pull yourself. Your AI assistant does it for you. Understanding the concept is enough.

.gitignore

When you push, does every file in your project folder need to go to GitHub? No. And you probably don't want it to.

There is a file called .gitignore. It lists the rules for which files should stay on your computer and not go online. A few common reasons why some files belong there:

  • Large data files. GitHub blocks files that are too big.
  • Temporary files and system junk, like the ones your Mac or Python creates in the background. They should not be uploaded.
  • API keys, passwords, anything private. These must never go online. .gitignore keeps them safe.
Important

Never commit secrets (API keys, passwords, .env files) to GitHub, even to a private repo. If you think you have, rotate the key immediately — once it's in Git history, removing it is complicated.

GitHub Pages

GitHub stores your code, and it can also serve your code as a webpage — for free, with a public URL anyone can open. That's GitHub Pages.

For each repository, there are two URLs you should know:

  • github.com/<your-name>/<repo> — this is the code.
  • <your-name>.github.io/<repo>/ — this is the webpage.

To make this work, we publish from a folder inside the repo called /docs/. That way your full project lives in the repo while only the web-ready files go online. Combined with your free Student Pro benefit, you can keep the code private while the webpage stays public.

Part 2

The edit loop: talking to your AI

Managing your GitHub Pages site is simple: you change anything on it just by talking to your AI assistant. No server to configure, no commands to memorize. Any language works, as long as you are clear.

Here is what it looks like in practice. You open Copilot Chat and type one message:

Change the subtitle of my webpage to 'Five Acts of European Democracy',
then commit and push.

Copilot finds where the subtitle is defined, edits the file, and uploads the change. You refresh your live page, and the subtitle has changed. You never type a Git command yourself.

Mental model

Think of Copilot as the person who knows Git. You tell it what you want changed; it handles how. The only skill you need is describing what you want clearly.

Part 3

Setup (one time per computer)

You only do this part once, the first time you use GitHub Pages on a given laptop. It installs a tool called gh (GitHub CLI) and connects it to your GitHub account.

Step 1. Configure Copilot Chat

Open Copilot Chat inside VS Code. Then:

  1. Click the model chip in the Copilot Chat input row and pick GPT-5.3-Codex. If Codex is rate-limited, any of these three backups works (all are agentic and can run commands):
    • Gemini 3.1 Pro (GitHub)
    • GLM 5.1 (OpenRouter)
    • Qwen 3.6 Plus (OpenRouter)
  2. Change Default Approvals to Autopilot (Preview) or Bypass Approvals, so Copilot can install things for you without stopping every few seconds.
If you don't see Autopilot

Your Copilot version is older. Pick Bypass Approvals instead — it does the same job.

Step 2. Run Prompt 1

Copy the whole text of Prompt 1, paste it into Copilot Chat, and send it. Copilot installs everything for you automatically.

Step 3. Authorize GitHub

During the process, Copilot will give you an 8-character code and open a browser tab. In that browser tab:

  1. Paste the 8-character code.
  2. Click Authorize.
  3. Make sure you allow all permissions — do not skip or deny anything, or the deploy step later will fail.

Once you return to VS Code, Copilot confirms everything is set up.

Allow all permissions

When the GitHub authorization page asks for permissions (repo access, workflow, etc.), grant them all. Partial permissions cause the deploy step to fail in confusing ways.

Manual fallback — if the browser tab doesn't open

Open github.com/login/device in any browser, sign in with your GitHub account, paste the 8-character code shown in your VS Code terminal, click Authorize, and allow all permissions. Then return to VS Code.

If Copilot never prints a code at all, open the VS Code terminal (View → Terminal) and run gh auth login --web by hand, then follow the prompts.

Troubleshooting Prompt 1

Prompt 1 installs gh through your operating system's package manager. If the package manager is missing, you can install gh manually.

Windows — winget not found

If you see an error saying winget is not recognized:

  1. Go to cli.github.com and download the .msi installer.
  2. Run the installer.
  3. Restart VS Code. If the new command is still not recognized, restart Windows as well.
  4. Re-run Prompt 1.
macOS — brew not found

If you see an error saying brew is not found, you have two options.

Option A (simplest, works for everyone). Go to cli.github.com, download the signed .pkg file, and install it directly. Restart VS Code, then re-run Prompt 1.

Option B (if you want Homebrew for other reasons). Go to brew.sh and follow their installation steps. Their instructions are clear, and Homebrew is a useful tool beyond this lab. Once Homebrew is installed, re-run Prompt 1.

Device code expired

If you do not finish the authorization within the time window, re-paste Prompt 1 to get a new code.

Part 4

Deploy your webpage

With setup done, deploying is one paste of Prompt 2. Copilot takes the folder you currently have open in VS Code, uploads it to a new GitHub repository, enables GitHub Pages, waits for the build, and gives you your live URL.

Step 1. Open the right folder in VS Code

Before running Prompt 2, make sure VS Code has the folder you want to publish. Use File → Open Folder and pick the folder that contains your index.html (or your project's main HTML file).

Common mistake

Opening the parent of your project folder, or the wrong project entirely. If Copilot says it can't find your files, this is almost always why.

Step 2. Paste Prompt 2 and send

Copy the whole text of Prompt 2, paste into Copilot Chat, and send. Copilot will:

  1. Check that gh is installed and you are logged in.
  2. Locate your webpage files and, if needed, copy them into a /docs/ folder. Your other files (.py, notebooks, data) are not moved.
  3. Create or update .gitignore with sensible defaults.
  4. Create a new public GitHub repository and push your files.
  5. Enable GitHub Pages from main branch, /docs folder.
  6. Wait for the build (up to ~3 minutes) and print the live URL.
  7. Write a new file called GITHUB_PAGES.md at your project root, summarizing everything it did.

If my demo deploys but your own project doesn't

This is the most common situation in the second half of the lab. Your setup is fine (the demo worked), but your project's folder structure is not quite what GitHub Pages or Prompt 2 expects. The fix is almost always to tidy up a few things before deploying.

What a well-structured project looks like

GitHub Pages serves static files. At minimum, there must be an index.html that the server can return when someone visits your URL. Here are three shapes that work cleanly:

Example 1. Minimal single page.

my-project/
└── index.html

Example 2. A full static site with assets.

my-project/
├── index.html
├── style.css
├── script.js
└── assets/
    ├── logo.png
    └── data.json

Example 3. A mixed project (data pipeline plus web output).

my-project/
├── dashboard.py            # generates the HTML
├── data/
│   └── raw.csv
├── docs/                   # the webpage Prompt 2 publishes
│   ├── index.html
│   ├── style.css
│   └── assets/
├── requirements.txt
└── README.md

In Example 3, your Python script stays at the root, your data stays in data/, and only the generated website lives under docs/. This is the pattern my demo uses, and it is the cleanest option when your project mixes code and web output.

A quick self-check before you run Prompt 2

Open your project folder in VS Code, look at the file tree on the left, and answer three questions:

  1. Is there a file named exactly index.html, either at the project root or inside a docs/ folder? The name must be lowercase and exact.
  2. Does that index.html reference other files (CSS, JS, images) that are actually in the project, at the paths it expects? Open index.html locally (double-click it in Finder or Explorer) and confirm the page renders correctly with images and styles. If it looks broken locally, it will look broken online.
  3. Do all your folder and file names use only letters, digits, hyphens, and underscores? No spaces, no Chinese characters, no special symbols. GitHub Pages URLs break on those.

If the answer to any of these is no, fix that first. Prompt 2 cannot publish what is not there.

The four most common structure problems

1. The main page is not called index.html. Rename home.html, main.html, or my-page.html to index.html. Without an exact match, GitHub Pages shows a 404 at your site root.

2. Links and image paths are broken. If index.html says <img src="/pictures/logo.png"> but your images actually live in assets/images/, nothing shows up. After deploying, open the live page, right-click, pick Inspect, and look at the Console tab for 404 errors. Each 404 tells you which path is wrong.

3. Your web files are in a folder Prompt 2 does not recognise. Prompt 2's whitelist is: assets/, images/, img/, css/, js/, fonts/, media/. If your project uses public/, web/, dist/, or something custom, either rename it to docs, or tell Copilot explicitly: "also copy the public folder into docs."

4. The folder is already a git repository pointing somewhere else. If your folder was cloned from a classmate or downloaded from a template, it may already have a .git directory connected to another remote. Prompt 2 refuses to overwrite it. Ask Copilot: "Detach the existing remote and start a fresh repository for me."

Asking Copilot to tidy up, step by step

A specific prompt works better than a vague one. Instead of "fix my project", paste this:

My project does not deploy cleanly with Prompt 2. Please do the following:

1. List every file and folder at my project root.
2. Identify which files belong in the published webpage (HTML, CSS, JS,
   images, fonts) and which do not (Python, data, notebooks, READMEs).
3. Propose a clean structure where the webpage files live inside docs/,
   and everything else stays where it is.
4. Show me the proposed moves in a list before changing anything.

Wait for my confirmation before doing any moves.

Review Copilot's proposal. If it looks right, say "yes, go ahead." If a specific move is wrong, point it out and ask for a revision. Then run Prompt 2 again.

Back up before any restructure

Before you let Copilot reorganise files, commit whatever you have (in the terminal: git add . && git commit -m "before restructure") or copy the whole folder to your Desktop as a safety net. If something goes wrong, you want a known-good state to return to.

Why this is fiddly the first time

Prompt 2 assumes a few conventions (an index.html at root or in docs/, standard web folder names, no colliding git history). Real student projects rarely match every assumption on the first try. Once you know the four problems above, fixing them takes a minute — the hard part is knowing what to look for.

If Copilot gets stuck mid-deploy

If Prompt 2 fails partway and can't recover even after pasting the error back to Copilot, see Part 7 · The fully manual path for step-by-step terminal commands and GitHub UI clicks that achieve the same result by hand.

Part 5

After your page is live

Updating your page

Once your page is online, do not re-run Prompt 2 to update it. The edit loop is simpler:

  1. Edit any file in your project.
  2. Ask Copilot in natural language, for example: "Commit and push." or "Change the subtitle to 'My Project Title' then commit and push."
  3. Wait about 30 seconds, refresh your live URL — the change is there.
Manual fallback

If you prefer the terminal, or if Copilot is down, open the VS Code terminal in your project folder and run:

git add .
git commit -m "your message here"
git push

Same result, no AI needed.

Making your repository private

By default, Prompt 2 creates your repository as public, which means your code is visible to anyone on the internet. For a lab submission or personal project, you probably want to keep the code private.

Just ask Copilot:

Please make my repository private.

Your webpage will keep working. This is because GitHub Pages visibility is independent of repository visibility — and your free GitHub Student Pro benefit includes private-repo Pages hosting.

Manual fallback

You can also do this from GitHub's website directly:

  1. Go to your repository page on github.com.
  2. Click Settings (top bar).
  3. Scroll all the way to the bottom, find the red Danger Zone section.
  4. Click Change repository visibility, then Make private.
  5. Type the repo name to confirm, and click the red confirmation button.

Your live URL continues to work because Student Pro unlocks Pages on private repos.

Before you submit

Unless you are explicitly okay with the world seeing your code (portfolios, public experiments), switch to private after deploying. Check your repository on github.com and confirm the private badge appears next to the name.

Deleting things safely

At some point you will want to delete something: an old file, an unused folder, or the whole repository. This is the single operation where it is easiest to lose work you did not mean to lose, so it is worth slowing down the first few times.

The core risk: when you ask Copilot (or run a git command yourself), a deletion often affects both your local computer and your online repository at once. A casual "delete the old-notes folder" can erase the folder from your laptop, from your git history, and from GitHub in one step, and there is no recycle bin to recover from.

Before you ask Copilot to delete anything:

  1. Keep a backup of anything you might still need. Copy the file to another location, or at minimum commit the current state first (git add . && git commit -m "snapshot before deleting X") so you can roll back.
  2. Be specific about scope. If you want the file gone from the online repository but kept on your laptop, say so: "remove X from the online repository only, keep my local copy untouched." If you want it gone everywhere, say "delete X from my project folder and from GitHub."
  3. Review Copilot's plan before it runs. Ask: "tell me exactly which files will be removed and from where, before you do it." Only approve once the list matches your intent.
Especially careful with

git rm -rf, rm -rf, "delete the whole repository", and anything that targets a directory by name. These remove entire trees in one shot and have no safe undo. If you need to empty a folder, consider moving it out of your project first (to your Desktop, for example), so you can put it back if something goes wrong.

Removing tracking, not the file

If you accidentally committed a file that should never have gone to GitHub (say, a large data file or a secret), you usually want to remove it from GitHub without deleting it locally. The correct command for that is git rm --cached <file>, followed by a commit and push. Ask Copilot: "remove X from git tracking but keep the local file, then push."

What is GITHUB_PAGES.md?

Prompt 2 writes a file called GITHUB_PAGES.md at the root of your project. It is a deployment log, not a README. It records:

  • Where the page lives (repo URL, live URL, Pages source).
  • What Prompt 2 changed (every action it took).
  • What Prompt 2 did not change (your original files, listed by name).
  • How to update the page later.
  • How to undo the deployment.

If anything about your deployment is unclear, open GITHUB_PAGES.md first — the answer is usually there. Your own README.md is untouched.

Part 6

If something goes wrong

Running into errors the first few times is normal. The combination of Copilot, GitHub, and whatever operating system you happen to be on is genuinely fiddly, and small differences in your environment (Windows vs. macOS, VS Code version, Copilot version, existing terminal state) surface in ways no tutorial can fully predict. Work through errors one at a time — each one gets you closer to a working deployment.

Universal fix

Whatever you see on screen — an error, something stuck, a command that didn't work — copy it, paste it into Copilot, ask it to fix it. Copilot knows what it just did, so it knows what went wrong. This handles almost every case.

The exceptions, where pasting the error to Copilot will not help:

  • A setting you cannot find in Copilot's own menu (for example, the Default Approvals dropdown does not appear). Copilot cannot see its own UI.
  • You are not sure you opened the right folder in VS Code, and Copilot keeps failing to find your files. Copilot cannot open folders for you.
  • Anything that blocks you for more than a few minutes even after pasting the error to Copilot. At that point, either switch to the manual path or email me.

Common issues at a glance

404 at your new URL

The build takes up to 3 minutes. Wait, refresh. If still 404 after 5 minutes, open the repo's Settings → Pages on github.com and confirm Source is main and /docs. Save again if needed.

Page loads but looks broken

Usually a path issue. Right-click the page, pick Inspect, open the Console tab, and look for red 404 errors. Each one points to a file the page tried to load but couldn't find.

"Repository name already exists"

Prompt 2 retries automatically with a date suffix (like -0423). If it doesn't, tell Copilot: "use a different repo name, add a date suffix."

Files you didn't expect in /docs

Your root has HTML files with non-standard names. Ask Copilot: "move the unnecessary files out of docs, keep only the webpage."

Expected files missing from /docs

Your project uses a subfolder Prompt 2's whitelist doesn't know. Ask: "also copy the <folder> folder into docs."

"Permission denied" when pushing

Your authorization is partial or expired. Re-run gh auth login --web in the terminal and allow all permissions this time.

"Not a git repository" error

Prompt 2 assumes your folder is ready to initialize. Ask Copilot: "initialise git in this folder and retry Prompt 2."

Changes don't appear after pushing

GitHub Pages caches aggressively. Wait 60 seconds, then do a hard refresh (Shift+Cmd+R on macOS, Ctrl+F5 on Windows). Also confirm the commit actually reached GitHub by opening your repo page.

Before emailing me

If an error is blocking you, try these in order: (1) paste the error to Copilot and ask it to fix it, (2) check the card above that matches your symptom, (3) look at Part 7 for the manual equivalent. If none of those solve it, email me a screenshot of the error and a brief description of what you were doing. Do not rerun Prompt 2 repeatedly — that can create duplicate broken repos.

Part 7

The fully manual path (if everything else fails)

If Copilot is completely down, your quota is exhausted, or you just want to understand what happens under the hood, this section walks through the entire deployment by hand — no AI involved. The commands below run in a terminal (inside VS Code: View → Terminal).

Only do this if needed

The AI-driven path is faster, safer, and less error-prone for most students. Use this manual path as a last resort, or to learn what is happening behind the scenes.

M1. Install gh (GitHub CLI)

macOS: download the signed .pkg from cli.github.com and double-click to install.

Windows: download the .msi from the same site, run it, and restart VS Code (and Windows, if the command is still not recognized afterwards).

Verify the install in a terminal:

gh --version

M2. Authenticate

gh auth login --web

Follow the prompts: pick GitHub.com, pick HTTPS, authorize via browser at github.com/login/device, allow all permissions.

M3. Prepare your project

In your project folder, create a docs/ folder and copy your web files into it:

mkdir -p docs
cp index.html docs/
# also copy any css, js, assets folders your page needs:
cp -r assets docs/ 2>/dev/null || true
cp -r css docs/ 2>/dev/null || true
cp -r js docs/ 2>/dev/null || true

Create a basic .gitignore (copy-paste the whole block):

.DS_Store
Thumbs.db
.vscode/
.idea/
__pycache__/
*.pyc
.venv/
.ipynb_checkpoints/
node_modules/
.env
*.key
*.pem

M4. Initialise git and commit

git init -b main
git add .
git commit -m "Initial deploy to GitHub Pages"

If git complains about missing identity:

git config user.name "your-github-username"
git config user.email "your-github-username@users.noreply.github.com"

M5. Create a GitHub repository and push

Replace my-project with whatever you want to call it:

gh repo create my-project --public --source=. --remote=origin --push

If you prefer to create the repo via the website: go to github.com/new, pick a name, make it Public for now, and click Create repository. Then, in the terminal:

git remote add origin https://github.com/your-username/my-project.git
git push -u origin main

M6. Enable GitHub Pages

From the command line:

gh api --method POST \
  repos/your-username/my-project/pages \
  -f "source[branch]=main" \
  -f "source[path]=/docs"

Or from the website:

  1. Open your repo on github.com.
  2. Click Settings → Pages (in the left sidebar).
  3. Under Source, pick Deploy from a branch.
  4. Set branch to main, folder to /docs, click Save.

M7. Wait for the build and visit your URL

GitHub needs up to 3 minutes to build the first version. Check status:

gh api repos/your-username/my-project/pages --jq .status

When it reports built, visit:

https://your-username.github.io/my-project/

M8. Make the repository private (optional)

gh repo edit --visibility private --accept-visibility-change-consequences

Or do it via the website: Settings → General → Danger Zone → Change visibility → Private. The live page keeps working thanks to Student Pro.

M9. Update the page later

Any time you want to publish a change:

git add .
git commit -m "describe what you changed"
git push

Wait ~30 seconds, refresh your live URL.

Stuck?

Even if you do this manually, you can still paste any error you see into Copilot (or any other AI) and ask for help. The manual path doesn't mean going it alone — it just means you are the one typing the commands.

Part 8

Going further

Once you have published one project this way, the same flow works for anything else with an index.html. A few ideas:

  • Your own personal website. A single HTML file, with a biography, some links, and maybe a photo. Put it on GitHub Pages and you have a permanent web address.
  • A class project portfolio. Put the visualizations and write-ups from each of your POLI courses into separate folders, each deployed the same way.
  • A collaborative research site. Keep the repo private, share read access with co-authors, and publish the front page as a public GitHub Pages site.
